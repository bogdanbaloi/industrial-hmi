# Performance profiling

CPU profiling of the headless console binary under a synthetic operator
workload, captured with Valgrind / callgrind, annotated to a stable text
report committed alongside the source. The discipline pairs with the
microbenchmarks under `benchmarks/`:

| Tool | Answers |
|---|---|
| `benchmarks/` (google/benchmark) | "How fast does *this function* run?" p50/p90/p99 of `AlertCenter::snapshot`, `decodeReadResponse`, `ConfigManager::initialize`. Microbench, no end-to-end context. |
| `scripts/perf/` (callgrind) | "Across a realistic session, where is the time going?" Whole-program inclusive cost per function. Spots the hot path even when no individual function looks slow on its own. |

Together: benchmarks pin the budget per primitive; the callgrind report
shows which primitives dominate the budget under a real workload.

## Why callgrind and not `perf`

`perf` is the natural choice for native CPU sampling, but:

1. **It needs kernel access.** `perf_event_open(2)` is gated by
   `kernel.perf_event_paranoid`. WSL2's default kernel does not enable
   the relevant counters, and CI runners typically refuse the syscall
   too. We refuse to ship a profiling story that only works on a
   sysadmin's workstation.
2. **It is a sampling profiler.** A 30-second workload that bursts for
   200 ms gets badly characterised by 99 Hz sampling. Callgrind counts
   every basic block executed and produces stable, deterministic output
   you can diff.
3. **We already use Valgrind in CI** (memcheck job on every PR). One
   more invocation is zero new infrastructure.

The trade-off is speed: callgrind is ~50x slower than native, so a
workload that runs in 2 seconds on metal takes ~100 seconds under
instrumentation. Fine for one-shot baseline capture; acceptable for an
on-demand CI job; not viable for continuous profiling.

## Files

| Path | Purpose | Committed? |
|---|---|---|
| `capture-callgrind.sh` | One-shot capture: runs callgrind on the console binary, pipes a workload through stdin, dumps `callgrind-annotate.txt`. | yes |
| `workload-baseline.txt` | Synthetic operator session: production lifecycle + equipment toggles + alarm snapshots + product reads + status loop. Chosen to exercise distinct hot paths rather than the same one repeatedly. | yes |
| `baseline.callgrind-annotate.top50.txt` | Trimmed top-50 inclusive-cost report from a reference run (Ryzen 7 5800X / WSL Ubuntu 24.04). The contract for regression diffing. | yes |
| `output/` | Per-run artefacts (`callgrind.out`, `console-stdout.txt`, full annotate). Regenerated by anyone re-running the script. | no (gitignored) |

## Running

```bash
# 1. Build the console binary (any preset, default release):
cmake --preset release
cmake --build build/wsl-release -j

# 2. Capture:
./scripts/perf/capture-callgrind.sh

# Expect ~30-60 s on the reference machine (workload runs once,
# instrumented). Top-15 functions print at the end; full annotate
# is written to scripts/perf/output/callgrind-annotate.txt.

# 3. Compare against the committed baseline:
diff -u scripts/perf/baseline.callgrind-annotate.top50.txt \
        <(head -90 scripts/perf/output/callgrind-annotate.txt)

# A change in the % column on the top-N rows is the signal. Don't
# look at absolute Ir counts -- they depend on the workload's exact
# byte stream + dynamic-linker symbol count + libc version.
```

To regenerate the committed baseline after a deliberate change:

```bash
head -30 scripts/perf/output/callgrind-annotate.txt \
    > scripts/perf/baseline.callgrind-annotate.top50.txt
grep -E '^\s*[0-9,]+\s+\(' scripts/perf/output/callgrind-annotate.txt \
    | head -60 \
    >> scripts/perf/baseline.callgrind-annotate.top50.txt
```

## How to read the baseline

The annotate file lists functions ranked by inclusive instruction count
(`Ir`) -- "how many basic blocks executed inside this function or any
function it called". The `%` column on the right is the share of the
program total.

Categories you will see in the captured baseline:

- **`./elf/dl-lookup.c` / `_dl_relocate_object` / `do_lookup_x`** (~30%
  combined) -- glibc dynamic linker resolving symbols on first call.
  Heavy because the binary links libsodium + sqlite + boost + open62541
  even without GTK. This is **startup cost**, not steady-state -- a
  longer workload amortises it down.
- **`./malloc/malloc.c::_int_malloc` / `_int_free`** (~5%) -- general
  allocator traffic. Heaviest under the JSON parser (string-key
  insertions) and the std::vformat logger.
- **`nlohmann::json::lexer::get()`** (~3%) -- config parse via
  `ConfigManager::initialize`. Matches the
  `bench_config_parse` baseline of ~42 us p50 on this hardware: that
  parse runs ONCE during bootstrap, and shows up here because the
  workload itself is short. In long-running operator sessions this
  percentage approaches zero.
- **`std::vformat`** (~1-2%) -- logger format-string lowering. Every
  `Logger::info(...)` call funnels through one non-template `vformat`.
- **`strcmp` / `memcpy`** (~3-5%) -- library internals; mostly libc /
  ld.so / SQLite.

What you should NOT see, and the signal flag if you do:

- Any of OUR functions above 5% inclusive. If `AlertCenter::raise` or
  `DashboardPresenter::handleNewWorkUnit` climbs into the top-10,
  something just regressed.
- Any virtual call dispatch (`__cxxabiv1::__vmi_class_type_info` and
  friends) above 1%. Indicates an O(N) loop hit a polymorphic call in
  the inner loop.
- `std::regex` anywhere. We do not use it on hot paths; appearance
  would mean a careless `regex_match` slipped in.

## Phase 2 (not in this commit)

- Graphical flame chart (`gprof2dot` + Graphviz `dot` -> SVG, OR
  Brendan Gregg's `flamegraph.pl` after a folded-stack conversion).
  Deferred because rendering tools (`graphviz`, `gprof2dot`) need
  apt / pip installs we haven't gated; the text-report path delivers
  the diagnostic value without that dependency.
- CI integration: optional `profiling` job that runs the capture on
  every PR and uploads the annotate artifact + a regression comment
  if the top-N % changes by more than a threshold.
- Per-subsystem workloads (`workload-historian.txt`,
  `workload-mqtt-loopback.txt`) so each backend gets its own
  baseline diffable across releases.
